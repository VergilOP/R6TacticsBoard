# Roadmap

本文档用于整理当前阶段之后的功能和结构推进方向，避免后续计划只散落在根 `README`，也避免在 `docs/` 下继续增加平行的计划文件。

## 当前阶段

当前仓库已经具备可用的桌面编辑器主链路：

- 编辑页支持 `单楼层` / `2.5D 总览` 双视图
- 2.5D 已接入轨道相机、全楼层播放和跨层额外过渡
- 电竞历史页已接入本地数据读取链路

接下来的工作重点不再是继续横向铺新页，而是先把编辑器底座、资源质量、交互手感和可扩展数据模型收口，再决定哪些新功能值得正式进入主界面。

## 优先顺序

建议按以下顺序推进：

1. 重建地图资源，先解决底图清晰度和坐标一致性
2. 暂时隐藏电竞功能入口，避免不稳定数据源分散主线
3. 优化编辑页的页面结构和操作反馈，优先提升流畅度、直觉性和细节体验
4. 增加额外调试模式，用于校准画布上的干员图标、名称和相关显示参数
5. 重构楼梯、舱口等跨层互动点模型，为路径规划和 2.5D 表达打底
6. 优化 `2.5D 总览` 的信息表达，降低“看得见但看不懂”的抽象感
7. 在稳定底座上引入统一的“战术标记”能力，再逐步承接人物元素与地图元素

## 重点方向

### 1. 地图资源重建

当前地图底图清晰度已经成为编辑体验和 2.5D 表达的共同瓶颈，应该优先处理。

实现思路：

- 重新下载和整理地图资源，优先保证主用地图的分辨率、命名和楼层对齐质量
- 保持 `map.json`、地图 key 和楼层 key 稳定，尽量不要让资源重建破坏现有工程文件引用
- 把“资源替换”和“坐标校准”视为同一项工作处理，不只换图，还要同步检查中心点、缩放感和 2.5D 层高参数
- 如有必要，补一个简单的资源校验或预览流程，避免后续继续把低清素材混入仓库

### 2. 电竞功能暂时收口

在没有可靠 API 和稳定更新链路之前，电竞页不应继续占据主界面心智。

实现思路：

- 暂时隐藏 `电竞历史` 页面入口，但保留现有代码和本地数据目录
- 优先用导航注册控制或简单开关控制页面显隐，而不是直接删除模块
- 相关数据读取层先保持最小维护状态，等待后续出现可靠数据源再继续扩展

### 3. 编辑页体验优化

编辑器接下来的目标应从“能编辑”提升到“顺手、流畅、符合直觉”。

实现思路：

- 梳理高频操作路径，例如加载地图、添加干员、选中、拖拽、切关键帧、播放、撤销/重做
- 优先优化操作反馈，包括选中态、悬停态、拖拽命中感、失败提示、当前上下文提示和快捷操作的一致性
- 检查页面布局密度，减少不必要的视觉噪音，让常用操作更近、低频功能更靠后
- 把体验优化视为持续收口工作，优先修“别扭感”和“认知中断点”，而不是一次性大改布局

### 4. 画布显示调试模式

当前需要一个额外的调试入口，用于校准干员在画布上的显示参数，而不是每次改代码后反复重启比对。

实现思路：

- 在调试页或编辑器调试浮层中增加一组专门面向画布显示的调试项
- 调试项重点覆盖干员图标大小、名称显示开关、名称大小、偏移、层级和不同缩放下的可读性
- 尽量让调试结果可以即时反映到当前画布和 `2.5D` 视图，方便肉眼比较
- 调试模式优先服务“快速试参数”和“确认视觉效果”，最终是否写回正式配置可以后置

### 5. 跨层互动点重构

当前把楼梯和舱口过度抽象成“一个点”的方式已经开始限制路径规划和表现力。

实现思路：

- 楼梯不再只表示为单点，而应支持“入口 / 出口”或“分段路径”这类更贴近实际移动的结构
- 舱口继续保留强约束的跨层连接，但在数据上也要明确源层、目标层和方向性
- 路径规划层与调试编辑层一起调整，避免只改渲染不改模型
- 调试页需要能清楚地区分不同互动点的连接关系、方向和有效楼层

### 6. `2.5D 总览` 表达优化

`2.5D` 的主要问题不是缺真 3D 模型，而是当前信息表达还不够清楚。

实现思路：

- 优先加强楼层分层感、当前路径高亮、互动点专用图元和干员/路径的可读性
- 通过透明度、描边、弱化非关注楼层、强化当前路径分段等方式降低画面混乱感
- 继续使用 2.5D 方案而不急着转向真 3D，把工作量集中在“更像可理解的战术视图”
- 地图级 `overview_2_5d` 参数仍然要保留，用于逐图修正层高、默认视角和绘制顺序

### 7. 统一战术标记系统

“干员技能/道具图标”与“封墙、开洞、翻越、对枪等地图元素图标”不应各做一套，应收敛为统一的场景标记能力。

实现思路：

- 先抽象通用标记模型，再区分人物元素与地图元素的分类、图标和附加属性
- 人物元素重点承接技能使用、道具携带、技能落点等信息
- 地图元素重点承接封墙、旋转洞、过人洞、翻越点、交火点等静态或半静态标记
- 标记系统初期应先支持放置、移动、删除、显示隐藏和基础序列化，避免一开始就接入过重的规则系统
- 后续若需要进入播放链路，再评估哪些标记是关键帧相关，哪些应作为全局战术注释存在

## 结构维护原则

- 新增功能优先落在现有分层目录中，不轻易再改顶层结构
- 优先扩展现有计划文档，不为每个想法单独新增一份路线文件
- 文档优先描述边界、顺序和实现思路，不提前写到实现细节失控
- 阶段变化时，优先同步根 `README` 与本路线文档，避免两边计划长期分叉
