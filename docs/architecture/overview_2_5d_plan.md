# 2.5D 全楼层总览方案

## 背景

当前编辑器一次只显示一个楼层。涉及垂直协同的战术时，用户需要频繁切换楼层，导致：

- 播放时难以连续理解上下层联动
- 垂直配合、舱口联动、楼梯转层不直观
- 同一帧的全楼层态势无法整体观察

因此需要补一个新的查看模式：`2.5D 全楼层总览`。

## 核心目标

- 在同一视图内同时展示整张地图的全部楼层
- 保留现有单楼层编辑模式，不改时间轴模型
- 重点服务播放、回看、理解垂直协同
- 操作方式接近 3D 游戏自由视角，底层用带贴图的楼层平面组织全楼层总览

## 当前状态

当前仓库已经完成第一阶段主干能力：

- 编辑页已支持 `单楼层` / `2.5D 总览` 切换
- 2.5D 使用 OpenGL 视图承载多楼层贴图平面
- 已支持左键 orbit、左右键同时按住平移、滚轮缩放、双击重置视角
- 播放时会展示全楼层状态，不再自动切换当前楼层
- 已支持 2.5D 下的干员 icon / 自定义名称显示
- 已支持 2.5D 下基于跨层路径的额外上下楼过渡时间

当前仍按“查看 / 播放模式”定位，不承担精细编辑。

## 交互要求

`2.5D 全楼层总览` 不使用顶部按钮调角度，改为鼠标驱动的轨道相机交互：

- 左键拖拽：绕中心点旋转观察角度
- 同时按住左右键拖拽：平移观察中心
- 滚轮：缩放
- 默认中心点：当前地图楼层堆栈的几何中心，或中间楼层附近的过渡高度

目标体验是“像转一个整体模型”，而不是“切预设角度”。

## 非目标

第一版暂不做：

- 真 3D 渲染
- 任意透视投影
- 2.5D 视图内直接编辑干员和互动点
- 替换当前底部悬浮播放条
- 调整时间轴结构

结论：`单楼层 = 编辑模式`，`2.5D = 观察 / 播放模式`。

## 现有实现限制

当前架构是明显的单楼层焦点模型：

- `EditorPage` 只有一个 `current_map_floor_key`
- 应用关键帧时会过滤非当前楼层状态
- 播放时会自动切换当前楼层
- `MapScene` 一次只加载一张楼层图

因此 `2.5D` 不能作为现有 `MapView` 的小补丁，而应是并行的新视图模式。

## 视图模式设计

编辑页增加显示模式切换：

- `单楼层`
- `2.5D 总览`

行为边界：

- `单楼层`：维持当前全部编辑、选择、楼层切换逻辑
- `2.5D 总览`：显示全部楼层，不自动把非当前楼层状态过滤掉
- 进入 `2.5D` 时，不改底部播放条布局和行为，只改画布层的渲染方式

## 渲染思路

### 总体方式

底层仍然是多张 2D 楼层图，但实现上会把它们作为带贴图的楼层平面放进 3D 视图容器。

每个楼层仍保留自身平面坐标：

- 地图上的 `x / y` 继续表示楼层内位置
- 新增一个逻辑高度 `z`
- 观察时把 `(x, y, z)` 投影到 2D 视口

这属于 2.5D，不需要建模数据。

### 相机模型

建议使用真正的 3D 视图相机，而不是写死角度：

- `azimuth / elevation`
- `distance`
- `center`

说明：

- 第一版优先开放 orbit / pan / zoom
- 默认中心点取地图中心 + 中间楼层高度

### 投影原则

- 所有楼层共用一套 3D 相机参数
- 楼层高度越高，离中心平面越远
- 相机旋转时，楼层平面和对象按真实 3D 视角观察
- 地图内对象和互动点使用同一套世界坐标

## 鼠标交互映射

建议的输入规则：

- 左键按下并拖动：orbit 相机
- 左右键同时按下并拖动：平移相机中心
- 滚轮：缩放 / 调整相机距离
- 双击或工具按钮：重置视角到默认中心和默认缩放

说明：

- 不建议首版做右键单独旋转，避免和现有上下文交互冲突
- 若 Qt 同时检测左右键不稳定，可接受“中键拖拽平移”作为技术回退方案，但目标仍是左右键组合平移

## 播放行为

`2.5D` 的重点是播放观感，因此播放逻辑要做以下切分：

- 单楼层模式：保持当前“跟随目标干员切层”的逻辑
- 2.5D 模式：关闭自动切层，直接渲染所有楼层状态

这意味着：

- `_state_matches_current_floor()` 在 `2.5D` 下不能再过滤
- `_sync_playback_focus_floor()` 在 `2.5D` 下应跳过
- 底部播放条保持原样，只驱动画布更新

## 互动点显示

### 舱口

- 在源楼层显示一个点
- 在目标楼层显示对应点
- 二者之间显示连接线
- 单向舱口可增加方向提示

### 楼梯

- 不建议简单把一个点连到多个楼层
- 渲染时应拆成楼层间分段
- 如果一个楼梯连三层，应显示两段连接关系

### 高亮

- 播放经过的互动点或跨层段可以高亮
- 未参与当前路径的互动点保持弱显示

## 当前实现边界

当前已经实现：

- 多楼层底图同时显示
- 2.5D 下的自由视角相机
- 楼层显隐面板
- 2.5D 干员 icon / 名称牌覆盖层
- 播放时的跨层路径总览
- 跨层播放的额外竖向过渡时间

当前还未完成：

- 2.5D 里针对互动点的独立高亮反馈
- 2.5D 里软墙 / `Hatch` 战术状态的同步表达
- 每张地图的 `overview_2_5d` 精调数据
- 2.5D 下直接编辑干员、互动点

## `v0.6.0` 收口结果

当前版本已经完成的范围：

- `2.5D` 已成为编辑页内的正式并行视图，而不是实验性按钮切换
- 全楼层贴图、轨道相机、楼层显隐、干员 icon / 名称牌、跨层路径总览已经接入同一条播放链路
- 跨层播放的额外上下楼时间已经生效，楼梯和舱口会在交互点位置产生额外竖向过渡

当前版本刻意没有做的内容：

- 不在 `2.5D` 里直接编辑干员与互动点
- 不改底部悬浮播放条整体布局
- 不把互动点单独做成完整的 3D UI 资产系统

## `map.json` 预留字段

为后续每张地图单独调优，建议在 `map.json` 中预留可选配置：

```json
"overview_2_5d": {
  "enabled": true,
  "default_yaw": 35.0,
  "default_zoom": 1.0,
  "pitch_factor": 0.65,
  "floor_height": 180,
  "draw_order": ["b1", "1f", "2f", "roof"],
  "floor_overrides": {
    "roof": {
      "height": 560
    },
    "b1": {
      "height": -180
    }
  }
}
```

楼层级也可预留：

```json
{
  "key": "2f",
  "name": "Second Floor",
  "image": "assets/maps/bank/2f.png",
  "overview": {
    "height": 360
  }
}
```

第一版可以没有这些字段，缺省时按楼层顺序自动推导。

## 推荐实现结构

建议新增并行组件，而不是重写现有单楼层画布：

- `presentation/widgets/canvas/overview_projection.py`
- `presentation/widgets/canvas/overview_scene.py`
- `presentation/widgets/canvas/overview_view.py`

职责建议：

- `overview_projection.py`
  负责地图坐标到 3D 世界坐标的换算
- `overview_scene.py`
  负责多楼层贴图平面、干员、互动点和路径的绘制
- `overview_view.py`
  负责 OpenGL 视图、鼠标轨道交互和相机控制

## 分阶段落地

### 第一阶段

- 增加 `2.5D 总览` 视图模式
- 实现全楼层底图同时显示
- 实现轨道相机式旋转 / 平移 / 缩放
- 播放时显示全楼层干员状态
- 2.5D 下显示干员 icon / 名称牌
- 跨层播放增加额外上下楼过渡
- 不改底部播放条

当前状态：已完成主干。

### 第二阶段

- 增加 `stairs` / `hatch` 专用 2.5D 图元与分段连线
- 增加互动点参与路径时的高亮反馈
- 增加地图级默认视角、层高和绘制顺序调优
- 增加常用地图的 `overview_2_5d` 参数回填

建议收口为 `v0.6.x`：

- 当前状态：`stairs` / `hatch` 的专用 2.5D 过层主链已接入，后续重点转向互动点高亮、软墙 / `Hatch` 状态和地图参数调优
- 先把互动点可视化补完整，再做各地图参数调优
- 保持 `2.5D = 播放 / 观察模式` 的边界，不在这一阶段扩大成完整编辑模式

### 第三阶段

- 进一步优化遮挡、选中高亮和跨层信息表达
- 评估是否需要 2.5D 下的轻量选中和只读信息提示
- 评估是否需要为电竞 /复盘模式复用 2.5D 相机和路径展示能力

建议放到 `v0.7.x` 之后评估：

- 是否复用 2.5D 相机给电竞复盘页
- 是否引入只读 hover 信息卡、目标跟随或镜头预设
- 是否需要 2.5D 下有限度的轻量交互

## 风险点

- 楼层图之间未必完全同尺度或同对齐
- 当前楼梯数据部分是“一点连多层”，展示时必须拆段
- Qt 下“左右键同时按住”的手势判定需要单独验证
- 如果未来要求在 `2.5D` 内直接编辑，复杂度会明显上升

## 建议结论

首版应严格收口为：

- 鼠标轨道相机式 `2.5D` 观察模式
- 服务播放和战术理解
- 不接管单楼层编辑
- 不改播放条布局

这样实现风险最低，也最符合本项目当前阶段需求。
